NPM Essentials

Posted on 2016-06-20   |   In Node.js   |  

NPM is a package manager for Node libraries.

Since NPM is part of the node.js, so you need to first install node.js in order to use it. Git Bash from Git is a good cli for using NPM for Windows user.

There are two ways to install modules in NPM.

  • install locally: npm install grunt-cli, which allows the package to be run in a specific application.
  • install globally: npm install -g grunt-cli, which allows the package to be run in any application.

Package.json is the file that keeps track of the modules in your project. You can use npm init to create this file.

After creating the json file, we can use the command line to add more modules.

  • Use npm install –save grunt-sass to add a production dependency module to the package.json file.

  • Use npm install –save-dev grunt-sass to add a development dependency module to the package.json file.

  • Use npm install to regenerate the node_modules folder.

The caret^ before version indicate you require the latest version of this application.

Here are a few more handy commands that we can use to manage our modules:

  • npm list lists all your packages.
  • npm -g list lists all your global packages.
  • npm outdated checks any outdated packages.
  • npm update updates any outdated package.
  • npm uninstall –save-dev grunt-sass uninstalls a package as well as gets rid of it from the package.json file.
  • npm prune cleans packages that are not listed in package.json file.

Bootstrap Grid System

Posted on 2016-05-25   |   In Bootstrap   |  

Bootstrap has been used widely today and its grid system is the core part of the framework.

The basic structure of the Bootstrap grid system.

First, we need to wrap rows and columns in a container. Bootstrap offers two types of container.

  • - full-width container that spans the entire viewport width.
  • - fixed-width container centered in the middle the viewport.

Inside the container, a row is used to contain all columns.

  • col-md-1 = 8.3% width
  • col-md-2 = 16.7% width
  • col-md-3 = 25% width
  • col-md-4 = 33.3% width
  • col-md-5 = 41.6% width
  • col-md-6 = 50% width
  • col-md-7 = 58.3% width
  • col-md-8 = 66.8% width
  • col-md-9 = 75% width
  • col-md-10 = 83.3% width
  • col-md-11 = 91.6% width
  • col-md-12 = 100% width

Bootstrap has several options for grid sizes:

xs - for the smallest screen widths like smartphones < 768 px
sm - for small screen widths like smartphones and tablets >= 544px
md - for medium screen widths like tablets and laptops >= 768px
lg - for large screen widths like desktops >= 992px
xl - for extra large screen widths like high-res desktops >= 1140px

The best part is we can combine the classes to make the columns look differently depending on screen size.

For the example above, col-md-3 would be 25% width on medium size screens, and the col-xs-6 class would make the same column 50% width on the smallest screens.

Another thing that is worth noticing is we can set a column hidden by using the .hidden-*-up or .hidden-*-down class. Here is a link for more info on the hidden classes.

How to use git

Posted on 2016-04-30   |   In Git   |  

Git is by far the mostly used version control software out there so it is crucial for us to have at least some basic understanding of it.

Let’s start by downloading a repository from GitHub.

  1. change to the directory where you want the cloned directory to be made
  2. use git clone https://github.com/YOUR-USERNAME/YOUR-REPOSITORY to clone a repository

Now you can see a cloned repository sitting in your local machine.

Navigating a Commit History

Here are some handy commands that you can use on the repository.

  • use git log to view all the commits in the repository.
  • use git log –stat to view which files are changed in all commits.
  • use git diff Older_commitID Newer_commitID to view the difference between two commits. (you may enter the first four characters of a commit ID instead the entire ID)
  • use git checkout commitID to roll back to a previous version.
  • use git status to see which files have changed since the last commit.

Creating and Modifiying a Repository

  1. change to the directory where you want to create a repository.
  2. use git init to create a repository.
  3. use git add fileName to add file to staging area.
  4. use git commit -m “commit message” to commit to the repository.

How the web works

Posted on 2016-04-28   |   In http   |  

This is a quick walk-through of how the web we use on a daily basis works.

In a short summary, there are three major parts that form the web, which are client, internet and server. The most frequent activity happens on the web is a client sends a request via internet to a server using HTTP protocol, then the server responds to the client by sending the requested file back to the client.

URL

The full name of URL is Uniform Resource Locator. Let’s analyze the components of a URL.

http://www.example.com:80/foo?p=1&q=neat#fragment

  • http is the protocol used for communication over internet
  • www.example.com is the host name, and an IP address is also a valid host name
  • :80 is the port, by default it’s 80 so you don’t need to specify in the URL
  • /foo is the path, which indicates what file you would like to request from the server
  • ?p=1&q=neat is the query parameter, which is the extra info get sent to the server when making a request
  • #fragment is not seen often, and it only exists on your browser and never get sent to a server.

HTTP

HTTP is in fact the main protocol of the web, which stands for Hyper Text Transport Protocol. Let’s get into it now.

Assume your browser is making a request to http://www.example.com/foo/logo.png?p=1#fragment.

The request begins with a request line:

GET /foo/logo.png?p=1 HTTP/1.1

  • GET is the method, indicating what type of request is making to the server.
  • /foo/logo.png?p=1 is the path, indicating what document you are requesting from the server. Pay close attention to the fact that query parameter is also sent to the server.
  • HTTP/1.1 is the version of the protocol, which most of the requests now speak.

Besides the request line, some headers are also get sent in the request like the following:

Host: www.example.com
User-agent: chrome

The reason that we need to include the host in the header is because one server may host multiple websites, so we need to specify which website we want to fetch the document. The user-agent describes who is making the request, giving the server extra info about the requester.

When the server hears your request, it would give your browser a response.

A typical response might look like this:

HTTP/1.1 200 OK, which is called a status line

  • HTTP/1.1 is the version of the protocol
  • 200 is the status code
  • OK is the reason phrase, this is nothing fancy but an English description of the status code

Some common status codes that worth remembering:

  • 200 OK, anything starts with 2 generally means ok
  • 302 Found, anything starts with 3 means the file is located somewhere else
  • 404 Not Found, anything starts with 4 means something wrong in the browser side
  • 500 Server Error, anything starts with 5 means something wrong in the server side

A number of headers followed the status line.

Date: Thu, 28 Apr 2016 01:28:12 GMT
Server: Apache/2.4.7 (Ubuntu)
X-Powered-By: PHP/5.5.9-1ubuntu4.13
Content-Length: 2631
Connection: close
Content-Type: text/html

Now let’s get into the playground to see how request and response works in practice.

  1. open your terminal (if you are using Windows you might need to download the telnet software)
  2. type telnet www.dennisxiao.com 80
  3. type GET /about.php HTTP/1.0
  4. type Host: www.dennisxiao.com
  5. hit enter to see the response from server

That’s it. This is how the web works!

Linux security

Posted on 2016-04-23   |   In Linux   |  

Today I have learned how to manage softwares, users and permissions on a Linux system to increase system security.

It is noticeable that when we log in our VM we are logged in as a standard user. As a standard user, you cannot execute some of the Linux commands. If you want to run some commands that are limited to system administrators, you can prepend the word ‘sudo’ at the beginning of your commands.

Command sudo ls -al /home/ubuntu/.ssh would list all files in the .ssh directory.

Software Update

Keeping softwares up-to-date plays a big part in system security. So now let’s come to the stage of updating and installing softwares on Linux. The following commands are useful in terms of installing and updating softwares on the system.

  • Use command cat /etc/apt/sources.list to get a list of software repositories.

  • Use command apt-get update to update the package source list.

  • Use command apt-get upgrade to update the softwares in the operation system.

(A handy shell trick is to combine the above two commands into sudo apt-get update && sudo apt-get upgrade)

  • Use command apt-get autoremove to remove anything that is useless in the system.

  • Use command apt-get install softwareName to install a software to the system.

  • Use command apt-get –purge remove softwareName to remove a software.

User Management

Another important file in the system is ‘/etc/passwd’, which keeps track of all the users in the system.

  • Use command cat /etc/passwd to peek inside the file.

Each row in the passwd file represents a user in the following format:

  • username:password:UID:GID:UID info:home directory:command/shell

User management plays an important role in terms of system security. Here are some commands related to user management.

  • Use command adduser userName to add a new user.

Once you have added a new user, you can use the following command to log in the system as the new user.

  • ssh userName@127.0.0.1 -p 2222

Let’s break this command down a bit.

ssh: ssh is the application we use to connect to a server
127.0.0.1: this is the IP we want to connect to
-p 2222: this is the port that we use to connect to the IP

Now we need to give the new user sudo access. Let’s assume we would like to create a new user named dennis.

  1. copy a current sudoer file and name it as the new user
    cp /etc/sudoers.d/vagrant /etc/sudoers.d/dennis

  2. edit the new user sudoer file using your preferred editor
    nano /etc/sudoers.d/dennis

  3. change the user info in the sudoer file
    dennis All=(All) NOPASSWD:ALL

After adding a new user, you can force the new user password to expire by using the following command:

  • passwd -e userName

Key-based Authentication

There is another authentication method, key-based authenticaton, that is more secure than using password.

The concept of a key-based authentication is that you generate a key pair on your local machine, and then you keep the private key in your machine and put the public key onto your server.

Here is how we generate a key pair locally.

  1. use ssh-keygen to generate a key pair
  2. save your key to /home/vagrant/.ssh/fileName
  3. now a key pair is generated, you should see in your specificed directory two new files, one private (key) and the other public (key.pub).

When you have generated a key pair, then you need to place the public key onto your server.

  • Use this handy one-line command cat ~/.ssh/id_rsa.pub | ssh user@hostname ‘cat >> .ssh/authorized_keys’ to push the key to a remote server.

Here are the detailed steps for how to putsh the key to a remote server:

  1. create a directory on your server using mkdir .ssh, .ssh is a special directory where all of your key related files must be stored.
  2. create a file using touch .ssh/authorized_keys, authorized_keys stores all the public keys that the current account allows to use for authentication.
  3. now go back to the local machine and copy the public key by using cat .ssh/key.pub.
  4. head to server and edit the authorized_keys file by using nano .ssh/authorized_keys and paste the public key into the file.
  5. set up file permission on the .ssh folder and authorized_keys file by using chmod 700 .ssh and chmod 644 .ssh/authorized_keys.

Now you can log in using key-based authentication.

  • Log in using ssh dennis@127.0.0.1 -p 2222 -i ~/.ssh/key

Yeah, now you have used key-based authentication instead of password. However, for greater security, we need to enforce all users log in using key-based authentication.

We need to edit the SSHD service config file to achieve that.

  • Use command nano /etc/ssh/sshd_config and change the PasswordAuthentication field to no.

When you are done with the configration change, you can restart the SSHD service by using service ssh restart.

File Permission

If you run a command like ls -l, the first column you see is the file permission information for a file or directory.

  • -rw-r–r– 1 vagrant vagrant 3637 Apr 9 2014 .bashrc

The first letter indicates if it is a file (-) or a directory (d). After that there are a total of nine ‘-‘, which can be separated into three groups.

owner group everyone
rwx | r– | r–

  • r indicates the read permission
  • w indicates the write permission
  • x indicates the execute permission

The second column is the owner and the third one is the group.

Now we need to dive a bit deeper into the chmod command.

Previously we used command like chmod 700 .ssh and chmod 644 .ssh/authorized_keys without really understanding what we are doing.

It turns out that Linux uses octal permission, so read, write and execute permissions can be expressed using numbers like the following:

  • r = 4
  • w = 2
  • x = 1

By adding the numbers together we will get the permission that we need to apply on a file or directory. For example, a read and execute permission will be 4 + 1 = 5.

Another thing that we need to remember is that for each file we need to give a three digits number, representing the permissions for owner, group and everyone respectively.

So now when we look back at our previous example of chmod 700 .ssh, we know that we give the owner all read, write and execute permissions, while stripping permissions all together from group and everyone.

We can also change owner or group permission for an entry using commands chown ownerName .bash_history or chgrp gruopName .bash_history.

Firewalls

Each application is configured to respond to requests destined for a specific port.

Here are a list of common ports that you would see quite offen:

21: File Transfer Protocol (FTP)
22: Secure Shell (SSH)
25: Simple Mail Transfer Protocol (SMTP)
53: Domain Name System (DNS) service
80: Hypertext Transfer Protocol (HTTP)
110: Post Office Protocol (POP3)
443: HTTP Secure (HTTPS)

We can use a firewall to configure which port we would like to listen to.

In Ubuntu, a firewall called ufw is pre-installed but by default it is not active.

  • Use command uwf status to check on the status of the firewall.

Because of the least privilege rule, we should start by blocking all our ports and then configure which port we would like to open. This practice would make managing rules much easier.

  • Use command uwf default deny incoming to block all the ports.
  • Use command uwf default allow outgoing to allow our server to send any request going out to internet.

Now it’s time to configure our firewall.

  • Use command ufw allow ssh to allow ssh.
  • Use command ufw allow www to allow http server.

After configuration done, we can turn on the firewall by using ufw enable.

Intro to Linux

Posted on 2016-04-23   |   In Linux   |  

When people say Linux is free, have you ever considered if it’s free as in beer or free as in speech? The success of Linux comes from its free as in speech approach. You can virtually run, see, modify and distribute the operation system however you would like.

Some sort of Linux knowledge is necessary if you want to get into web development job. So let’s dive right into the Linux world!

Before we go deeper, it’s handy to know some Vagrant commands.

Previously we have learned to use ‘vagrant up’ to fire up the virtual machine and ‘vagrant ssh’ to get into secured shell. Here are some other commands that are useful.

  • ‘vagrant status’ is used to check on the status of the VM.
  • ‘vagrant suspend’ is used to suspend your VM, which means to put it into a sleep mode.
  • ‘vagrant halt’ is used to turn off your VM. (let’s call it a day and power off your viftual machine!)
  • ‘vagrant destroy’ is used to format your VM, and to roll it back to baseline Linux installation.

Now we are in the shell. It’s good to have some basic understanding of the important folders in your VM.

If you go to your home directory and type ‘ls -al’, a detailed list of directories should show up. Some of the folders here are really important and we need to get acquaintance with them.

  • ‘etc’ directory is where the configuration files live
  • ‘var’ directory is where you can find system and application logs
  • ‘bin’ directory is where executable binaries are stored
  • ‘sbin’ directory is where applications for system administrator sit, normal users typically don’t use

Here you may wonder why we can simply type ‘ls’ if all the commands live in ‘bin’ directory instead. This is where $PATH comes into play.

Try ‘echo $PATH’ to see your current PATH.

If you haven’t fiddled with your PATH setting much then you would expect to see something like this

/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/usr/games:/usr/local/games

When you type a command, the Linux would travel trough these directories one by one to see if the command is in there. You can edit this path at /etc/environment if it need be.
