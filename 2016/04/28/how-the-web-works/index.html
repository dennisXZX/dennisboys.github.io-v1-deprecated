How the web works

Posted on 2016-04-28   |   In http   |  

This is a quick walk-through of how the web we use on a daily basis works.

In a short summary, there are three major parts that form the web, which are client, internet and server. The most frequent activity happens on the web is a client sends a request via internet to a server using HTTP protocol, then the server responds to the client by sending the requested file back to the client.

URL

The full name of URL is Uniform Resource Locator. Let’s analyze the components of a URL.

http://www.example.com:80/foo?p=1&q=neat#fragment

  • http is the protocol used for communication over internet
  • www.example.com is the host name, and an IP address is also a valid host name
  • :80 is the port, by default it’s 80 so you don’t need to specify in the URL
  • /foo is the path, which indicates what file you would like to request from the server
  • ?p=1&q=neat is the query parameter, which is the extra info get sent to the server when making a request
  • #fragment is not seen often, and it only exists on your browser and never get sent to a server.

HTTP

HTTP is in fact the main protocol of the web, which stands for Hyper Text Transport Protocol. Let’s get into it now.

Assume your browser is making a request to http://www.example.com/foo/logo.png?p=1#fragment.

The request begins with a request line:

GET /foo/logo.png?p=1 HTTP/1.1

  • GET is the method, indicating what type of request is making to the server.
  • /foo/logo.png?p=1 is the path, indicating what document you are requesting from the server. Pay close attention to the fact that query parameter is also sent to the server.
  • HTTP/1.1 is the version of the protocol, which most of the requests now speak.

Besides the request line, some headers are also get sent in the request like the following:

Host: www.example.com
User-agent: chrome

The reason that we need to include the host in the header is because one server may host multiple websites, so we need to specify which website we want to fetch the document. The user-agent describes who is making the request, giving the server extra info about the requester.

When the server hears your request, it would give your browser a response.

A typical response might look like this:

HTTP/1.1 200 OK, which is called a status line

  • HTTP/1.1 is the version of the protocol
  • 200 is the status code
  • OK is the reason phrase, this is nothing fancy but an English description of the status code

Some common status codes that worth remembering:

  • 200 OK, anything starts with 2 generally means ok
  • 302 Found, anything starts with 3 means the file is located somewhere else
  • 404 Not Found, anything starts with 4 means something wrong in the browser side
  • 500 Server Error, anything starts with 5 means something wrong in the server side

A number of headers followed the status line.

Date: Thu, 28 Apr 2016 01:28:12 GMT
Server: Apache/2.4.7 (Ubuntu)
X-Powered-By: PHP/5.5.9-1ubuntu4.13
Content-Length: 2631
Connection: close
Content-Type: text/html

Now let’s get into the playground to see how request and response works in practice.

  1. open your terminal (if you are using Windows you might need to download the telnet software)
  2. type telnet www.dennisxiao.com 80
  3. type GET /about.php HTTP/1.0
  4. type Host: www.dennisxiao.com
  5. hit enter to see the response from server

That’s it. This is how the web works!
